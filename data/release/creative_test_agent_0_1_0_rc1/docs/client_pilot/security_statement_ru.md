# Закрытый контур обработки данных

## Что означает закрытый контур

Архитектура пилота спроектирована для локальной обработки без облачных LLM/API. Все данные — креативы, бренд-профили, аудиторные профили, результаты тестов и отчёты — остаются на компьютере пользователя. Система не отправляет данные на внешние серверы для анализа.

## Какие данные обрабатываются локально

- Тексты креативов и загруженные файлы (txt, md, pdf, png, jpg, webp).
- Профили бренда (ton of voice, restrictions, claims policy).
- Профили целевой аудитории (сегменты, боли, мотивации, возражения).
- Результаты тестирования (оценки по критериям, риски, рекомендации).
- Сгенерированные отчёты (Markdown, HTML).
- Логи аудита (события системы).
- Метаданные визуального анализа (при его выполнении).

## Какие внешние LLM не используются

- OpenAI (GPT-4, GPT-3.5)
- Anthropic (Claude)
- Google (Gemini)
- Perplexity
- Любые другие облачные LLM-провайдеры

Локальный LLM-слой поддерживает:
- **Stub** (режим по умолчанию) — детерминированные ответы, без внешних вызовов.
- **Ollama** (опционально) — локальный запуск моделей через Ollama.
- **LM Studio** (опционально) — локальный запуск моделей через LM Studio.

## Какие cloud OCR/Vision API не используются

- Google Vision API
- AWS Textract
- Azure OCR
- OpenAI Vision
- Gemini Vision
- Claude Vision
- Любые другие облачные Vision/OCR API

Визуальный анализ (опционально) использует:
- **Stub** (режим по умолчанию) — детерминированные ответы.
- **Local OCR** (Tesseract) — опционально, при установленном Tesseract.
- **Local VLM** (Ollama vision model) — опционально, при настроенной Ollama.

## Где хранятся файлы

Загруженные файлы сохраняются в локальную файловую систему в директории, указанной в `CTA_STORAGE_ROOT` (по умолчанию `./data/storage`). Путь к файлам хранится в базе данных, но сами файлы не копируются в облачные хранилища.

## Где хранится база данных

По умолчанию используется SQLite (`creative_test_agent.db`) в корне проекта. База данных не передаётся по сети и не реплицируется в облако. При необходимости может быть использована PostgreSQL (локальная или внутри инфраструктуры клиента).

## Как проверяется отсутствие cloud-зависимостей

Проект включает встроенные проверки:

```bash
make check-closed-loop
```

Скрипт `scripts/check_closed_loop.py` проверяет:
1. Отсутствие установленных cloud SDK (`openai`, `anthropic`, `google-cloud-vision`, `boto3`, `azure-ai-vision`).
2. Корректность переменных окружения (`CTA_LOCAL_ONLY_MODE=true`, `CTA_ALLOW_CLOUD_LLM=false`).
3. Безопасность LLM и Vision провайдеров.

Полная проверка готовности:

```bash
make pilot-check
```

## Какие ограничения есть у MVP

- SQLite не рассчитан на высокую конкурентную нагрузку (для пилота достаточно).
- Локальная LLM (Ollama/LM Studio) требует установки дополнительного ПО.
- Локальный OCR (Tesseract) требует установки Tesseract OCR engine.
- Безопасность на уровне пилота: нет разграничения прав, API-ключей или production-аутентификации.
- При использовании stub-режима выводы основаны на детерминированных правилах, а не на анализе LLM.
